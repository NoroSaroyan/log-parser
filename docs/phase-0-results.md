## Три категории потерь

### 1. Полностью отброшенные массивы шагов (в БД их нет)

Сюда попадает:
- `Skipping unmatched step array(s) — more step arrays than station records` (диспатчер Phase 0) — лишние массивы, которые не с чем парить в рамках группы;
- `Test steps cannot be matched to any test station` (парсер) — массивы, для которых вообще нет ни одной StationInformation-записи с этим PCBA (напр. `T32444111` в 20260415).

| Файл | Массивов | Тест-шагов |
|---|---:|---:|
| 20260411 | 1 | 38 |
| 20260415 | 3 (2 excess + 1 orphan `T32444111`) | 141 (94 + 47) |
| 20260416 | 15 | 579 (14×38 + 1×47) |
| 20260417 | 5 | 190 (5×38) |
| **Итого** | **24 массива** | **~948 шагов** |

Доля — **0.73 %** от всех 129 983 шагов в БД. На уровне тест-сессий **24 из ~2271** (~1 %).

### 2. Записаны в БД, но под станцией не того типа (не потеряны, но мислейблены)

Phase 0 WARN `Dispatcher paired step array to station of different type`. Массив шагов паровался по позиции с записью станции другого типа — данные вставились, но FK ведёт на «не ту» станцию.

| Файл | PCBA | PCBA/Final путаница |
|---|---|---|
| 20260411 | `H8444A11100T32343298` | PCBA-шаги → Final-запись |
| 20260411 | `H8444A11100T32645715` | Final-шаги → PCBA-запись (retry с лишней станцией) |
| 20260416 | `H8444A11100T32443984` | PCBA-шаги → Final-запись |
| 20260417 | `H8444A11100T33147078` | PCBA-шаги → Final-запись |

Итого **4 пары**, примерно **~180 шагов** лежат в БД под неверным родителем. Данные есть, их не нужно перекачивать — после Phase 1 спаривание будет по `(PCBA, StationType)`, при повторном прогоне эти записи перепишутся корректно.

### 3. Непарсимо в принципе

`steps_missing_pcba_scan` — массивы шагов без `PCBA Scan` / `Compare PCBA Serial Number` / `Valid PCBA Serial Number`: 2 в 20260411, 1 в 20260415, итого **3 массива**. Это устройства, которые отвалились раньше, чем станок успел считать PCBA — таких шагов в принципе не с чем соединить. Ни наш фикс, ни заводской это не восстановит.

## Что закроет Phase 1

- **Retry-асимметрия** (например, `T32646747`: 1 Final-запись + 2 Final-массива): сейчас второй Final-массив отбрасывается. Phase 1 через FIFO-спаривание по `(PCBA, StationType)` сохранит оба массива — либо привяжет ко второй Final-записи если она есть, либо, если её реально нет в логе, **синтезирует** запись с `is_synthetic=true`.
- **Bug #1** (например, `T32343298`: есть Final-запись, но нет PCBA-записи, при этом шаги PCBA-этапа пришли): сейчас PCBA-шаги привязываются к Final-записи (мислейбл из категории №2). Phase 1 **синтезирует** PCBA-запись, заимствуя `LogisticData` у Final-сестры, шаги ложатся под правильную станцию.
- **Мислейблы из категории №2** исчезают как явление: пара всегда будет вычисляться по `(PCBA, StationType)`, не по позиции в слайсе.
- Остаётся только категория №3 (устройства, упавшие до чтения PCBA) — это заводской вопрос, Phase 1 на них не влияет.

Ожидаемое состояние после Phase 1: из 24 отброшенных массивов сохранится **23** (все кроме `T32444111` — у него нет ни одной StationInformation, заимствовать `LogisticData` не у кого, поэтому остаётся orphan). Мислейблы обнулятся.
