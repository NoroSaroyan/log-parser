# Разбор отличий между версиями тестового ПО и их влияние на наш парсер

**Цель документа.** Сравнить имеющиеся «старые» рабочие логи (`corporate_resources/old_logs/`, версии ПО станков `2.0.1` и `2.0.3`) с текущими «новыми» проблемными логами (`corporate_resources/*.gz`, версия `2.0.11-RC`), зафиксировать **поведенческие отличия** между версиями и показать, **какие именно из этих отличий** вступают в конфликт с логикой нашего `log-parser` и приводят к сбоям обработки. Документ предназначен для внутреннего ревью команды и планирования фикса на нашей стороне.

Мы не оцениваем качество обновления поставщика оборудования — задача в другом: описать то, что изменилось, на пальцах, с цитатами из логов, и указать на те изменения, которые **наш** парсер не умеет обрабатывать в текущей своей форме.

---

## 2. Отличие №1 — словарь имён `TestStepName`

### Описание

Список уникальных значений `TestStepName` в payload'ах `POST /v1/testdatas` расширился:

| Версия | Уникальных имён |
|---|---:|
| `2.0.1` | 54 |
| `2.0.3` | 59 |
| `2.0.11-RC` | 65 |

### Конкретные изменения

Шесть имён, присутствовавших в `2.0.1`, **отсутствуют** в `2.0.11-RC`:

```
Delay 3 Seconds
Disable Security Access
Get TCU Product Date
MES Check
Waiting For Modem Initialized
Write TCU Product Date
```

Десять имён присутствуют **только** в `2.0.11-RC` (и в `2.0.3`):

```
Get Function Configuration Code
Get TCU Production Date
Get TSP Registration Status
KL15 Off
Set TSP State To Be Registered
Stm Variable DID Fields Check
Wait For Modem Initialization To Complete
Write Function Configuration Code
Write TCU Part Number
Write TCU Production Date
```

Из них три — переименования (одно и то же действие, новое имя):

- `Get TCU Product Date` → `Get TCU Production Date`
- `Write TCU Product Date` → `Write TCU Production Date`
- `Waiting For Modem Initialized` → `Wait For Modem Initialization To Complete`

Остальные семь — **новые шаги тестовой последовательности** (TSP-регистрация, запись part number, управление питанием KL15, проверка DID-полей, запись Function Configuration Code).

### Влияние на наш парсер

**Нулевое.** Наш парсер привязан только к трём именам `TestStepName`: `PCBA Scan`, `Compare PCBA Serial Number`, `Valid PCBA Serial Number` (это скан-шаги, из которых мы извлекаем номер PCBA и определяем тип сессии). Эти три имени сохранились неизменными во всех версиях. Остальные шаги для парсера — просто строки, которые сохраняются в `test_step.test_step_name` как есть.

Это изменение **не ломает** нас. Фиксить ничего не надо.

## 3. Отличие №2 — `POST /v1/testdatas` без сопутствующего `POST /v1/stationinformation`

### Описание

Для части устройств станок отправляет `POST /v1/testdatas` с массивом тест-шагов, но **не отправляет** соответствующий `POST /v1/stationinformation` (с метаданными сессии: тип станции, тестер, PCBA, LogisticData, время завершения теста).

### Количественно

| Подтип | `2.0.1`–`2.0.3` (7 файлов) | `2.0.11-RC` (4 файла) |
|---|---:|---:|
| Пришли PCBA-шаги, нет PCBA-записи | 0 | 3 |
| Пришли Final-шаги, нет Final-записи | 0 | 6 |
| Пришли шаги, нет ни одной записи (orphan) | 0 | 1 |

Суммарно в 4 новых файлах — 10 затронутых устройств из ~2247. На 7 старых файлах такого — **ноль**.

### Пруф на конкретном устройстве: `H8444A11100T32343298`, файл `mesrestapi.log-20260411.gz`

Шаги PCBA-станции пришли (цитата из лога, строки 470008–470014):

```
470008: Apr 10 12:09:35 ... TestDatas Serving: /v1/testdatas from mesrestapi.pandora.pri
470009: Apr 10 12:09:35 ... Data  [
470010: Apr 10 12:09:35 ...  {
470011: Apr 10 12:09:35 ...    "TestStepName": "PCBA Scan",
470012: Apr 10 12:09:35 ...    "TestThresholdValue": "",
470013: Apr 10 12:09:35 ...    "TestMeasuredValue": "H8444A11100T32343298",
```

Все `POST /v1/stationinformation` в этом файле для данного PCBA:

```
Apr 10 17:48:54 ... Inserting StationInformation:  {0 703003734AA Final Tester_01 ... H8444A11100T32343298 ...}
```

Одна запись, тип `Final`. Записи типа `PCBA` в файле нет вообще (`grep 'Inserting StationInformation' | grep T32343298 | grep -c 'PCBA'` → `0`).

Другие примеры того же подтипа (`2.0.11-RC`):

- `H8444A11100T33147078` (файл `20260417`) — PCBA-шаги 06:21, запись только Final в 13:04.
- `H8444A11100T32443984` (файл `20260416`) — PCBA-шаги есть, запись только Final.

Зеркальный подтип (нет Final-записи, но есть Final-шаги):

- `H8444A11100T32444508` (файл `20260416`) — есть PCBA-запись и PCBA-шаги в 08:00, в 14:23 приходит `POST /v1/testdatas` с шагом `Valid PCBA Serial Number` для этого же PCBA, но `POST /v1/stationinformation { TestStation: "Final", ... }` не приходит.

Подтип «orphan — нет ни одной записи»:

- `H8444A11100T32444111` (файл `20260415`) — в 15:04 приходит `POST /v1/testdatas` с `PCBA Scan`, и больше в файле ничего от этого устройства нет.

### Влияние на наш парсер

**Критичное.** Наш парсер сопоставляет массив шагов со станцией по лукапу в карте `allStations[pcba]`:

```go
// internal/services/parser/json_parser.go, второй проход
for _, testStepData := range testStepsToProcess {
    ...
    if _, ok := allStations[pcbaFromSteps]; ok {
        results = append(results, steps)
    } else {
        // WARN и пропуск
    }
}
```

Карта индексируется **только по номеру PCBA** — тип станции не учитывается. Когда у устройства в логе есть запись **одного** типа (например, Final), а шаги — и PCBA, и Final, все массивы шагов успешно проходят проверку «ключ в карте есть» и попадают в `results`. На этапе группировки группа для этого PCBA получает: 1 запись станции и 2 массива шагов. Ниже по пайплайну в диспатчере:

```go
// internal/services/dispatcher/dispatcher.go, итерация шагов
for i, stepsSlice := range group.TestSteps {
    if i >= len(testStationIDs) {
        return fmt.Errorf("mismatch in TestSteps count and TestStationRecords count ...")
    }
    ...
}
```

— раньше (до Phase 0 hotfix) `return` прерывал обработку **всего файла** на первом же таком устройстве. После Phase 0 hotfix (`commit 1f654ed`) — логируется WARN, и «лишние» массивы шагов не вставляются; остальные устройства в файле продолжают обрабатываться, но связанные с этим PCBA данные теряются частично. В БД для такого устройства окажется одна запись станции (например, Final) с привязанными шагами PCBA-этапа — семантически неверно (Phase 1 это исправит штатным спариванием по `(PCBA, StationType)`).

На `2.0.1` / `2.0.3` ни одного случая такого поведения не наблюдалось, поэтому слабое место парсера не задевалось.

## 4. Отличие №3 — асимметрия количества `POST`'ов при повторных прогонах

### Описание

Устройство прогоняется по одной и той же станции несколько раз (повторный тест после сбоя или доработки оператора). На `2.0.1` / `2.0.3` каждому `POST /v1/testdatas` от станка соответствовал отдельный `POST /v1/stationinformation` — пары были строгими. На `2.0.11-RC` число пар для одного устройства одного типа станции расходится: для одного и того же PCBA приходит, например, **1 `stationinformation`** и **2 `testdatas`**, или **3 `stationinformation`** и **4 `testdatas`**.

### Количественно

| Тип асимметрии | `2.0.1`–`2.0.3` (7 файлов) | `2.0.11-RC` (4 файла) |
|---|---:|---:|
| На одном устройстве одного типа `N stationinformation ≠ M testdatas`, тип PCBA | 0 | 4 |
| То же, тип Final | 1 | 13 |

В старых логах — один случай за 7 файлов (единичный, ближе к шуму). В новых — 17 случаев на 4 файла.

### Пруф на конкретном устройстве: `H8444A11100T32645382`, файл `mesrestapi.log-20260415.gz`

Три `POST /v1/stationinformation { TestStation: "PCBA" }` для этого PCBA в файле:

```
Apr 14 05:44:09 ... Inserting StationInformation:  {0 ... PCBA ... 2026-04-14 05:44:08 ... false P013 ... H8444A11100T32645382 ...}
Apr 14 06:43:58 ... Inserting StationInformation:  {0 ... PCBA ... 2026-04-14 06:43:58 ... false P013 ... H8444A11100T32645382 ...}
Apr 14 06:49:05 ... Inserting StationInformation:  {0 ... PCBA ... 2026-04-14 06:49:04 ... true  ...     H8444A11100T32645382 ...}
```

Три попытки: две зафейленные (`P013`), третья passed — классический retry.

При этом `POST /v1/testdatas` с PCBA Scan для того же PCBA в этом файле — **четыре** (по количеству прогонов + один, не привязанный к записи). Итого 3 записи станции + 4 массива шагов того же типа.

Ещё два характерных примера (`2.0.11-RC`):

- `H8444A11100T32646747` (файл `20260416`) — одна `PCBA`-запись, одна `Final`-запись, но при этом **один** массив PCBA-шагов и **два** массива Final-шагов. Final-тест прошёл дважды (08:17 и 16:07), но `POST /v1/stationinformation` типа `Final` пришёл один раз.
- `H8444A11100T32646421` (файл `20260415`) — одна `PCBA`-запись (`P013`, fail) и два массива PCBA-шагов.

В файле `20260416` этот паттерн самый частотный — **12 устройств** из 15 mismatch'ей имеют форму `1 Final stationinformation + 2 Final testdatas`.

### Влияние на наш парсер

**Критичное.** Наш парсер держит станции и массивы шагов в **параллельных срезах**:

```go
// internal/domain/models/dto/GroupDataDTO.go
type GroupedDataDTO struct {
    DownloadInfo       DownloadInfoDTO
    TestStationRecords []TestStationRecordDTO
    TestSteps          [][]TestStepDTO
}
```

Диспатчер сшивает их **по индексу**: `TestSteps[i]` должен пойти под `TestStationRecords[i]`. При равенстве `len(TestSteps) == len(TestStationRecords)` и корректном порядке в логе это работает. Как только счёт расходится:

- если `len(TestSteps) > len(TestStationRecords)` — срабатывает проверка `i >= len(testStationIDs)` в `dispatcher.go`, возвращается ошибка (после Phase 0 hotfix — WARN + пропуск «лишних» массивов);
- если `len(TestSteps) < len(TestStationRecords)` — лишние записи станций остаются без шагов, молча. Данные технически сохранены, но часть тест-шагов реальных retry-прогонов потеряна.

Сшивка по индексу также неявно полагается на совпадение **типа** в каждой паре. Когда у одного PCBA в группе приходят, например, 1 PCBA-запись + 1 Final-запись + 1 PCBA-массив + 1 Final-массив, в индексной сшивке `PCBA[0] ↔ PCBA_steps[0]` и `Final[1] ↔ Final_steps[1]` — но порядок в слайсе не гарантирован. Если `results` парсера случайно положил Final-запись первой, шаги PCBA-этапа уйдут под Final-станцию.

На `2.0.1` / `2.0.3` оба инварианта (равенство счёта и согласованный порядок) соблюдались, поэтому парсер работал корректно без их явного enforcement'а.

## 5. Отличие №4 — `StationInformation` с пустым `PCBANumber`

### Описание

В `2.0.11-RC` в некоторых `POST /v1/stationinformation` payload'ах приходит пустой `"PCBANumber": ""` при заполненном `"ProductSN"`. Обычно это устройства, которые зафейлились на ранней стадии теста (до чтения PCBA-серийника) — например, с кодом ошибки `F202`, `F065`.

### Количественно

| Событие | `2.0.1`–`2.0.3` | `2.0.11-RC` |
|---|---:|---:|
| `StationInformation` с пустым `PCBANumber` | 0 (плюс 1 в `20250927`) | 3 (2+1+0+0) |

### Пример: файл `20260411`

```
Apr 10 06:24:32 ... Data {
  "TestStation": "Final",
  "IsAllPassed": false,
  "ErrorCodes": "F202",
  "LogisticData": {
    "PCBANumber": "",
    "ProductSN": "YCOT1EBG30900FB#",
    ...
  }
}
```

### Влияние на наш парсер

**Минорное.** Группировщик в `internal/services/processor/grouper.go` при пустом `PCBANumber` делает фолбэк на `ProductSN`:

```go
case dto.TestStationRecordDTO:
    key := strings.TrimSpace(v.LogisticData.PCBANumber)
    if key == "" {
        key = strings.TrimSpace(v.LogisticData.ProductSN)
        ...
    }
```

Такая запись уходит в БД с `logistic_data.pcba_number = ProductSN` — то есть в таблице появляется строка, где значение `pcba_number` на самом деле — ProductSN, а не PCBA. Обработка не падает, данные формально вставляются, но отчётность по `pcba_number` начинает содержать «фантомные» значения вида `YCOT1EBG...#`.

Phase 1 предлагает пропускать такие записи с WARN, чтобы не засорять `logistic_data` (или переложить в отдельную таблицу `early_failures` — решение в `docs/fix-plan.md`).

## 6. Что не изменилось между версиями

Здесь важно явно зафиксировать: мы проверили эти пункты и подтвердили, что они **не** являются источником сбоя.

- **Значения `TestStation`**: ровно три во всех 11 файлах — `Download`, `PCBA`, `Final`. Никаких новых типов.
- **Набор полей в `POST` payload'ах**: полностью совпадает с нашими DTO (`DownloadInfoDTO`, `TestStationRecordDTO`, `LogisticDataDTO`, `TestStepDTO`). Ни одного переименованного ключа.
- **Структура вложенности**: `LogisticData` остался вложенным объектом внутри `TestStationRecord` payload. Всё остальное — плоские поля.
- **Порядок событий в логе**: `POST /v1/stationinformation` и `POST /v1/testdatas` по-прежнему идут во временной близости друг к другу; в `2.0.11-RC` просто иногда одного из двух нет, но если оба есть — они расположены так же, как раньше.
- **Типы значений**: `TestMeasuredValue` по-прежнему может быть строкой или числом (`interface{}` в DTO).

## 7. Какие из отличий ломают наш парсер

| # | Отличие | Ломает нас? | Где именно в коде |
|---|---|---|---|
| 1 | Словарь `TestStepName` (ре­неймы + новые шаги) | Нет | — |
| 2 | `testdatas` без пары `stationinformation` | **Да, критично** | `json_parser.go`: `allStations` индексируется по PCBA без типа; `dispatcher.go`: сшивка по индексу приводит к mismatch или неверной привязке |
| 3 | Асимметрия пар при retry-прогонах | **Да, критично** | `GroupedDataDTO`: параллельные срезы `TestStationRecords` / `TestSteps` без явного enforcement'а их парности; `dispatcher.go`: сшивка по индексу |
| 4 | `StationInformation` с пустым `PCBANumber` | Нет, но засоряет БД | `grouper.go`: фолбэк на `ProductSN` |

**Главный вывод:** отличия №2 и №3 — это то, что «убило» обработку. До `2.0.11-RC` наш парсер работал в предположении, что:

- для каждого PCBA и каждого типа станции в логе есть **либо** полная пара `stationinformation` + `testdatas`, **либо** ничего;
- число пар по одному типу совпадает с числом massive шагов этого типа для одного устройства.

Оба предположения в `2.0.11-RC` перестали выполняться (отличия №2 и №3). Наш код нигде эти предположения явно не enforced: они были **неявным контрактом** между форматом лога и парсером. Как только реальный лог начал от них отходить, парсер стал валиться.

## 8. Как это исправить на нашей стороне

План фикса зафиксирован в `docs/fix-plan.md`, детально с цитатами и примерами — в `docs/problem.md`. Ключевое:

- **Phase 0** (уже выкатано, `commit 1f654ed`). Диспатчер больше не роняет файл при mismatch'е группы — логирует WARN и продолжает. Это немедленный workaround: из 4 проблемных файлов теперь в БД попадают все ~2247 групп, вместо нуля. Но частично теряются данные у ~24 устройств, и ~4 пары стоят под записью станции не того типа.

- **Phase 1** (структурный фикс). Переход с параллельных срезов на модель «сессия = станция + её шаги». Парсер сопоставляет массивы шагов со станциями по паре `(PCBA, StationType)`, а не только по PCBA. При отсутствии парной `stationinformation` (отличие №2) — синтезируем минимальную запись с `is_synthetic=TRUE`, заимствуя `LogisticData` у соседней сессии того же PCBA, чтобы не терять тест-шаги. При retry-асимметрии (отличие №3) — FIFO-сшивка сессий по порядку появления в логе. Phase 1 закрывает оба критичных отличия и снимает зависимость от неявного 1:1-контракта.

- **Phase 2–6** (валидатор, миграция БД под идемпотентность и `is_synthetic`, транзакции, тесты, сводки) — укрепляют систему, чтобы следующее аналогичное расхождение формата не привело к новому пожару.

## 9. Воспроизведение и проверка

Скрипт, собирающий этот анализ — `/tmp/compare_old_new.py`. Запуск из корня репозитория:

```bash
python3 /tmp/compare_old_new.py
```

Обрабатывает все файлы в `corporate_resources/old_logs/*` и `corporate_resources/mesrestapi.log-*.gz`, выдаёт:
